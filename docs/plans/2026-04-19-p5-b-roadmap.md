# P5.B OSC Dispatch Roadmap

## Goal

Turn `P5.B` into bounded OSC stream-dispatch slices instead of treating all OSC
semantic routing as one broad task.

## Upstream scope

- `upstream/ghostty/src/terminal/stream.zig`
  - `oscDispatch`
- `upstream/ghostty/src/terminal/osc.zig`

## Dependency notes

- `P5.B` depends on the current translated `osc.Command` surface.
- `P4.C` and `P4.D` can still extend `osc.Command` later. When they do, `P5.B`
  must grow with them through new bounded sub-slices rather than reopening a
  vague umbrella implementation turn.
- The remaining slices should follow upstream `stream.zig` `oscDispatch`
  action families rather than parser-phase labels alone, so each task can land
  as a coherent stream-facing change.
- Once the corresponding parser outputs exist, `P5.B3+` are worker-sized
  slices, but they currently share the same write set
  (`src/terminal/stream.mbt`, `src/terminal/stream_test.mbt`,
  `src/terminal/pkg.generated.mbti`), so they should still execute serially
  unless OSC stream dispatch is split further first.

## Completion criteria for P5.B

`P5.B` is complete only when all of the following are true:

- all `P5.B*` sub-slices are `done`
- every translated `osc.Command` variant is either:
  - routed to a typed `StreamAction`, or
  - intentionally silent because upstream stream behavior is logging-only
- latest `P5.B` code slice passes:
  - `moon check`
  - `moon test`
  - `moon coverage analyze`
  - `moon fmt`
  - `moon info`
- coverage residuals in touched files are either eliminated or explicitly
  accepted in the slice audit
- `.mbti` public-surface diffs are reviewed and intentional
