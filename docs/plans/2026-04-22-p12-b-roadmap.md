# P12.B Roadmap

## Goal

Split the input-encoder work into green commit boundaries that match the real
translation risk:

- `P12.B1`: `mouse_encode.zig`
- `P12.B2`: `key_encode.zig`

## Why the split is necessary

The original single `P12.B` row hid two very different workloads.

- `mouse_encode.zig` is self-contained. It depends on existing mouse enums,
  input events, terminal query state, and byte encoding helpers.
- `key_encode.zig` pulls in a much larger protocol surface: Kitty key tables,
  PC-style function-key tables, control-sequence rules, modify-other-keys
  behavior, and macOS option-as-alt handling.

Treating both as one task would force a broad mixed commit and make coverage
and public-API review harder than necessary.

## MoonBit surface policy

- Do not reproduce the C out-buffer API.
- Use MoonBit-owned encoder objects with typed setters/getters and `Bytes`
  results.
- Keep the real encoding logic internal to the package unless an external
  caller needs a separate stable abstraction.

## Task order

1. `P12.B1`: land the mouse encoder owner surface and tests.
2. `P12.B2`: land the key encoder owner surface and tests.

## Acceptance notes

- `P12.B1` should cover terminal-derived defaults, format selection,
  out-of-viewport behavior, and motion dedupe/reset.
- `P12.B2` should cover terminal-derived defaults plus at least the upstream
  Kitty/legacy cases needed to make the public encoder surface credible for
  downstream runtime work.
