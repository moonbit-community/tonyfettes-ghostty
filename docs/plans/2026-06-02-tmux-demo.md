# Mini Tmux PTY Example

## Goal

Add a small native MoonBit example under `examples/tmux` that demonstrates a
minimal tmux-like multiplexer built with `moonbit-community/pty`,
`moonbit-community/tty`, and the translated Ghostty terminal surface. The demo
must not shell out to the system `tmux` binary.

## Accepted Design

- Add an executable package at `examples/tmux`.
- Add `moonbit-community/pty@0.2.0`, `moonbit-community/tty@0.2.4`, and
  `moonbitlang/async@0.19.1` to the examples module.
- Use `@tty.Tty` to own the outer terminal raw mode, alternate screen, key
  events, resize events, and redraws.
- Start with one shell pane backed by `@pty.Pty`.
- Add prefix-key commands:
  - `Ctrl-b c` creates a new shell pane.
  - `Ctrl-b n` and `Ctrl-b o` switch to the next pane.
  - `Ctrl-b p` switches to the previous pane.
  - `Ctrl-b b` sends a literal `Ctrl-b` to the active pane.
  - `Ctrl-b q` and `Ctrl-b x` quit the demo.
- Feed each pane's PTY output into its own `@terminal.StreamTerminal`.
- Forward ordinary keyboard input to the active pane's PTY.
- Render the panes with a simple native MoonBit layout and status line.
- Connect `StreamTerminal` query replies back to each pane's PTY through a small
  local reply buffer, because `StreamTerminal::write_pty` is synchronous while
  `Pty::write` is async.
- Keep the example package native-only; the existing Rabbita examples remain
  JS-only.

## Target Files and Surfaces

- `examples/moon.mod`
- `examples/tmux/moon.pkg`
- `examples/tmux/main.mbt`

No `src/terminal` implementation or public API change is expected.

## API / Interface Diff

- New executable command: `moon -C examples run --target native tmux`
- `src/terminal/pkg.generated.mbti` should remain unchanged.

## Why Existing Code Cannot Be Reused As-Is

- `tools/stream_terminal_perf` feeds synthetic byte streams and does not own a
  PTY.
- `tools/terminal_playground_core` is a wasm-facing playground and only records
  terminal query replies; it cannot call async native PTY APIs.
- The Rabbita asciinema example replays recorded terminal output rather than
  spawning a live terminal program.
- The previous sketch that launched the system `tmux` binary demonstrated PTY
  capture, but it did not implement a multiplexer and depended on an external
  program.

## Open Questions

- Runtime success depends on a POSIX-like shell being available as `sh`; the
  example is native-only and Unix-oriented because it owns PTYs and a raw TTY.

## 2026-06-02 Server/Client Checkpoint

### Goal

Split the mini tmux demo into a single-binary server/client architecture so a
client can detach without killing pane shells, while keeping the demo native,
small, and independent of the system `tmux` binary.

### Accepted Design

- Keep one executable package at `examples/tmux`.
- Use the same binary in two modes:
  - default mode is the client, which owns the user's raw TTY and alt screen.
  - `--server` mode owns sessions, panes, PTYs, and terminal models.
- On client startup, read a JSON discovery file for the current user's mini
  tmux server. If connecting fails, treat the file as stale and start a new
  server.
- Start the server by spawning the current executable as an orphan process in
  `--server` mode, following the self-executable/self-argv pattern used by
  `moonbit-community/pty`.
- Preserve the current native runner argv when spawning from `moon run`, and
  use an internal `MOONBIT_MINI_TMUX_SERVER_DIR` environment marker to enter
  server mode. Under `moon run`, the executable path may be the native runner
  rather than the final app binary, so spawning `self_executable --server`
  directly is not reliable.
- Use a package-local native stub to detach the server process on Unix
  (`fork`, `setsid`, `/dev/null` stdin, log-file stdout/stderr, then `execvp`).
  `moonbitlang/async/process.spawn_orphan` intentionally does not change the
  Unix process session, so it is not sufficient for tmux-style detach.
- Use localhost TCP on all platforms for the first server/client protocol. The
  server binds `127.0.0.1:0` and writes the OS-assigned port into discovery
  state.
- Parse the executable CLI with `moonbitlang/core/argparse`, including internal
  `--server` and `--state-dir` options plus user-facing attach and kill-session
  commands.
- Store the discovery file under a user-local state directory:
  - Linux/Unix: `$XDG_RUNTIME_DIR`, then `$XDG_STATE_HOME`, then
    `$HOME/.local/state`.
  - macOS: `$HOME/Library/Application Support`.
  - Windows: `%LOCALAPPDATA%`.
  - fallback only when needed to the platform temp directory.
- Use a stable `server.lock` file for discovery synchronization. Do not lock
  `server.json` itself, because `server.json` is replaced atomically through a
  temp-file-then-rename write path.
- Keep discovery data as JSON with `version`, `state`, `pid`, optional `port`,
  and `created_at_unix_ms`. `state` is `starting` while a newly spawned server
  is binding its socket and `running` after it publishes the assigned port.
- Do not use an authentication token for the demo protocol. The discovery file
  would have to contain the same token, so it would not materially protect
  against same-user local processes that can read the file. State directory
  permissions and the local-only random TCP port are the intended demo boundary.
- Keep the protocol simple for this demo: client sends input, resize, detach,
  kill-pane, and kill-session commands; server sends full text frames for
  redraws.
- Move pane ownership and `StreamTerminal` parsing to the server. The client
  only renders server frames to the outer TTY and sends local input/events.
- Align prefix keys with tmux-like semantics:
  - `Ctrl-b d` detaches the client.
  - `Ctrl-b x` kills the active pane.
  - `Ctrl-b Q` kills the whole demo session.
  - `Ctrl-b q` is reserved for pane display/selection and should no longer
    quit the demo.

### Why Existing Code Cannot Be Reused As-Is

- The current mini tmux is single-process; the raw-TTY client loop owns the
  same task group as the pane PTYs, so exiting the UI necessarily tears down
  panes.
- `moonbit-community/pty` owns PTY child spawning but does not provide a
  server/client control plane or persistent session discovery.
- `moonbitlang/async/socket` already provides portable TCP support, so the first
  cross-platform demo should use localhost TCP before adding Unix sockets or
  Windows named pipes.

### Open Questions

- `Pty::close` still uses the default `@process.spawn` graceful cancellation
  path, which can wait up to five seconds for interactive shells that ignore
  SIGTERM. A separate `moonbit-community/pty` API change to expose
  `cancel_handler` or hard cancellation would make kill-pane/kill-session
  faster and cleaner.
- Multi-session naming is deferred; this checkpoint implements a single default
  session.

### Next Implementation Step

Refactor `examples/tmux/main.mbt` so the server owns panes and rendering state,
the client owns local TTY state, and both communicate over a small localhost TCP
protocol discovered through the JSON state file.
